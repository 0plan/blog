# Git vs SVN: 분산형과 중앙집중형 버전 관리 시스템 비교

소프트웨어 개발에서 버전 관리 시스템(VCS)은 필수적인 도구입니다. 현재는 **Git**이 사실상 표준으로 자리 잡았지만, 여전히 많은 기업에서 **SVN(Subversion)**을 사용하고 있습니다. 이 두 시스템의 근본적인 차이점과 각각의 장단점을 정리해 보겠습니다.

---

## 1. 핵심 아키텍처의 차이

가장 큰 차이점은 데이터가 저장되고 관리되는 방식, 즉 **중앙집중형(Centralized)**과 **분산형(Distributed)**의 차이입니다.

### **SVN: 중앙집중형 (CVCS)**
*   **구조:** 하나의 중앙 서버에 전체 프로젝트의 히스토리가 저장됩니다.
*   **방식:** 개발자는 서버에서 특정 버전의 코드(Working Copy)만 내려받아(Check-out) 작업하고, 변경 사항을 다시 서버에 보냅니다(Commit).
*   **특징:** 히스토리를 확인하거나 커밋을 하려면 반드시 **중앙 서버와 연결**되어 있어야 합니다.

### **Git: 분산형 (DVCS)**
*   **구조:** 서버뿐만 아니라 모든 개발자의 로컬 PC에 전체 프로젝트의 히스토리가 통째로 복제되어 있습니다.
*   **방식:** 로컬에서 커밋, 로그 확인, 브랜치 생성 등을 자유롭게 수행하고, 필요할 때만 서버와 동기화(Push/Pull)합니다.
*   **특징:** 서버가 없어도 대부분의 작업을 **오프라인**으로 수행할 수 있으며, 서버 장애 시에도 로컬 복제본으로 복구가 가능합니다.

---

## 2. 데이터 저장 방식: Delta vs Snapshot

### **SVN: Delta 방식**
SVN은 파일의 기본 버전과 그 이후의 **차이점(Delta)**을 저장합니다. 특정 버전의 파일을 만들려면 초기 버전부터 모든 차이점을 순차적으로 적용해야 하므로, 히스토리가 길어질수록 속도가 느려질 수 있습니다.

### **Git: Snapshot 방식**
Git은 파일을 차이점이 아닌 **스냅샷(Snapshot)**으로 관리합니다. 커밋할 때마다 파일 시스템 전체의 상태를 사진 찍듯 저장하며, 변경되지 않은 파일은 새로 저장하지 않고 이전 파일에 대한 링크만 유지합니다. 이 방식은 버전 전환과 비교 연산이 매우 빠릅니다.

## 3. 브랜치(Branch)와 머지(Merge)

| 구분 | Git | SVN |
| :--- | :--- | :--- |
| **브랜치 생성** | 단순히 커밋을 가리키는 포인터 생성 (매우 빠름) | 물리적인 디렉토리 복사 (비교적 무거움) |
| **머지(Merge)** | 커밋 단위로 변경 사항을 자동 추적 (강력함) | 파일 단위로 차이점을 수동에 가깝게 계산 (복잡함) |
| **워크플로우** | 기능별 브랜치 생성이 일상적임 | 브랜치 생성을 가급적 자제하는 경향 |

---

## 4. 주요 비교 요약

| 특징 | Git (분산형) | SVN (중앙집중형) |
| :--- | :--- | :--- |
| **속도** | 로컬 작업이므로 매우 빠름 | 네트워크 속도에 의존적임 |
| **오프라인 작업** | 가능 (커밋, 로그, 브랜치 등) | 불가능 (체크아웃된 코드 수정만 가능) |
| **보안/권한** | 전체 저장소를 복제하므로 부분 권한 제어가 어려움 | 디렉토리별로 세밀한 권한 제어가 가능함 |
| **학습 곡선** | 명령어가 많고 개념이 복잡하여 다소 높음 | 구조가 직관적이고 단순하여 낮음 |
| **대용량 파일** | 바이너리 파일 처리에 취약할 수 있음 (LFS 필요) | 대용량 바이너리 파일 관리에 유리함 |

## 5. 왜 Git을 사용해야 하는가? (Why Git?)

### **① 폭발적인 생산성과 오프라인 자유도**
Git은 모든 히스토리를 로컬에 가지고 있습니다. 커밋, 로그 조회, 코드 비교, 브랜치 전환이 서버 통신 없이 **0.1초 내외**로 끝납니다. 비행기 안에서나 인터넷이 불안정한 곳에서도 모든 버전 관리 기능을 100% 활용할 수 있습니다.

### **② 브랜치 중심의 개발 문화 (Feature Branch Workflow)**
Git의 브랜치는 단순히 특정 커밋을 가리키는 포인터일 뿐입니다. 새로운 기능을 만들 때마다 브랜치를 만들고, 실험하고, 실패하면 바로 버리는 과정이 매우 가볍습니다. 이는 현대적인 **Agile 방식**과 **CI/CD 파이프라인** 구축에 필수적인 요소입니다.

### **③ 막강한 생태계와 협업 도구**
GitHub, GitLab, Bitbucket 등 전 세계적인 협업 플랫폼은 모두 Git 기반입니다. 수많은 오픈 소스 라이브러리와 연동 가이드가 Git을 표준으로 삼고 있으며, 문제가 발생했을 때 StackOverflow 등에서 해결책을 찾기도 훨씬 쉽습니다.

## 6. 왜 SVN을 사용해야 하는가? (Why SVN?)

### **① 직관적인 중앙 통제와 보안**
SVN은 중앙 서버가 진실의 유일한 원천(Single Source of Truth)입니다. 관리가 명확하며, 특히 **디렉토리/파일 단위의 세밀한 접근 권한 제어**가 가능합니다. 특정 팀에게는 특정 폴더만 보이게 하거나 쓰기 권한을 주는 기능은 Git에서 기본적으로 지원하지 않는 SVN만의 강력한 장점입니다.

### **② 대용량 파일(Binary) 관리의 효율성**
Git은 전체 히스토리를 복제하므로 고해상도 이미지, 영상, 설계도면 같은 대용량 바이너리 파일이 많아지면 로컬 저장소 용량이 순식간에 수십 GB로 늘어납니다. 반면 SVN은 현재 필요한 버전만 내려받아 작업하므로, 디자인 자산이나 리소스가 많은 **게임 개발, 디자인 프로젝트**에서 여전히 선호됩니다.

### **③ 낮은 학습 곡선과 단순한 워크플로우**
Git의 `rebase`, `cherry-pick`, `reflog` 같은 개념은 초보자에게 매우 혼란스럽습니다. 반면 SVN은 `update`와 `commit`이라는 단순한 개념만으로도 충분히 운영이 가능하므로, 비개발 직군(디자이너, 기획자)과 협업할 때 훨씬 수월합니다.

## 7. 어떤 프로젝트에 어떤 도구가 어울릴까?

| 프로젝트 유형 | 추천 도구 | 이유 |
| :--- | :--- | :--- |
| **현대적 웹/앱 개발** | **Git** | 빠른 배포 주기가 필요하고, 브랜치 활용이 많으며, 오픈소스 라이브러리를 많이 사용함. |
| **오픈 소스 프로젝트** | **Git** | 전 세계 개발자들이 각자 Fork하여 작업하고 Pull Request를 보내는 분산 협업에 최적화됨. |
| **게임 개발 (리소스 중심)** | **SVN** | 대용량 3D 모델, 텍스처, 사운드 파일이 많아 전체 히스토리를 복제하기 부담스러움. |
| **금융/정부 보안 프로젝트** | **SVN** | 폴더별로 엄격한 접근 권한 제어가 필요하고, 작업자의 로컬에 전체 소스 코드가 남는 것을 방지해야 함. |
| **단순 문서/보고서 관리** | **SVN** | 버전 관리 개념이 생소한 사용자들에게 복잡한 Git 명령어보다 직관적인 사용성을 제공함. |
