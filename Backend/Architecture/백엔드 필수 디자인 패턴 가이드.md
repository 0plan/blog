# 백엔드 필수 디자인 패턴 가이드: 예제로 배우는 효율적인 설계

디자인 패턴은 소프트웨어 개발 과정에서 반복적으로 발생하는 문제들에 대한 '검증된 해결책'입니다. 백엔드 개발자에게 디자인 패턴은 코드의 재사용성을 높이고 유지보수 비용을 낮추는 강력한 무기가 됩니다.

---

## 1. 생성 패턴 (Creational Patterns)

객체 생성 로직을 캡슐화하여 시스템의 유연성을 높이는 패턴입니다.

### **① 싱글톤 패턴 (Singleton)**
*   **설명:** 클래스의 인스턴스를 단 하나만 생성하고 어디서든 그 인스턴스에 접근할 수 있게 합니다.
*   **DB 관련 예시:** **Database Connection Pool (HikariCP)**. DB 연결은 비용이 매우 크기 때문에, 커넥션 풀 객체를 싱글톤으로 만들어 애플리케이션 전체에서 공유합니다.
*   **코드 예시:**
    ```java
    @Configuration
    public class DbConfig {
        @Bean
        @Primary // 단 하나의 DataSource만 생성하여 공유
        public DataSource dataSource() {
            return new HikariDataSource();
        }
    }
    ```

### **② 빌더 패턴 (Builder)**
*   **설명:** 복잡한 객체의 생성 과정과 표현 방법을 분리합니다.
*   **DB 관련 예시:** **QueryDSL / jOOQ 쿼리 생성**. 복잡한 SQL 쿼리를 자바 코드로 조립할 때 빌더 패턴이 사용됩니다.
*   **코드 예시:**
    ```java
    // QueryDSL 예시: 빌더 방식으로 쿼리를 조립
    List<User> users = queryFactory
        .selectFrom(user)
        .where(user.age.gt(18))
        .orderBy(user.name.asc())
        .fetch();
    ```

## 2. 구조 패턴 (Structural Patterns)

### **① 어댑터 패턴 (Adapter)**
*   **설명:** 호환되지 않는 인터페이스를 중간에서 연결해 줍니다.
*   **DB 관련 예시:** **JDBC (Java Database Connectivity)**. 자바는 하나인데 DB는 MySQL, Oracle, PostgreSQL 등 다양합니다. 각 DB 드라이버가 JDBC라는 표준 인터페이스에 맞춰 '어댑터' 역할을 하기 때문에, 개발자는 DB가 바뀌어도 같은 자바 코드를 쓸 수 있습니다.

### **② 프록시 패턴 (Proxy)**
*   **설명:** 실제 객체 접근 전후에 로직을 추가합니다.
*   **DB 관련 예시:** **JPA 지연 로딩 (Lazy Loading)**. 엔티티를 조회할 때 연관된 데이터를 즉시 가져오지 않고, 실제 사용할 때까지 '가짜 프록시 객체'를 세워둡니다. 실제 데이터를 요청하는 순간 프록시가 DB에 쿼리를 날려 데이터를 채워옵니다.

---

## 3. 행위 패턴 (Behavioral Patterns)

### **① 전략 패턴 (Strategy)**
*   **설명:** 알고리즘을 동적으로 교체합니다.
*   **DB 관련 예시:** **다중 DB 라우팅 (Master/Slave 분기)**. 서비스 규모가 커지면 쓰기 작업은 Master DB로, 읽기 작업은 Slave DB로 보내야 합니다. 이때 현재 트랜잭션의 상태에 따라 DB 연결 전략을 선택합니다.
*   **코드 예시:**
    ```java
    public class RoutingDataSource extends AbstractRoutingDataSource {
        @Override
        protected Object determineCurrentLookupKey() {
            // 현재 트랜잭션이 ReadOnly면 SLAVE, 아니면 MASTER 반환
            return TransactionSynchronizationManager.isCurrentTransactionReadOnly() 
                   ? "SLAVE" : "MASTER";
        }
    }
    ```

## 4. 백엔드 아키텍처 패턴 (Architectural Patterns)

### **리포지토리 패턴 (Repository Pattern)**
*   **설명:** 도메인 모델과 DB 저장소 사이의 중재자 역할을 합니다.
*   **DB 관련 예시:** **Spring Data JPA Repository**. 서비스 레이어에서는 이 인터페이스만 바라보고, 데이터가 실제 MySQL에 있는지 메모리(H2)에 있는지는 신경 쓰지 않습니다. 
*   **이유:** 테스트 시에는 가짜 DB(Mock)를 주입하고, 운영 시에는 실제 DB를 주입하여 코드 수정 없이 환경을 전환할 수 있습니다.

---

## 5. 요약 및 주의사항

1.  **패턴은 목적이 아닌 수단입니다:** 코드를 패턴에 억지로 끼워 맞추는 '오버엔지니어링'을 경계해야 합니다.
2.  **가독성과 유지보수:** 패턴을 쓰는 이유는 나중에 코드를 고치기 쉽게 하기 위해서입니다.
3.  **스프링과의 조화:** 자바 백엔드 개발자라면 스프링과 JPA가 내부적으로 어떤 패턴을 사용하여 DB 통신을 효율화하는지(Connection Pool, Proxy, Repository 등) 이해하는 것이 가장 중요합니다.
