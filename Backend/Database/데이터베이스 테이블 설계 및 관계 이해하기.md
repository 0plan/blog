# 데이터베이스 테이블 설계 및 관계 이해하기 (PK, FK, Unique)

데이터베이스 설계의 시작은 테이블을 만들고 그들 사이의 관계를 정의하는 것입니다. 단순히 데이터를 넣는 곳을 넘어, 데이터의 무결성을 지키고 효율적으로 조회하기 위한 핵심 개념들을 정리합니다.

---

## 1. 테이블 제약 조건 (Constraints)

데이터가 정확하고 안전하게 보관되도록 강제하는 규칙들입니다.

### **① Primary Key (PK, 기본키)**
*   **역할:** 각 행(Row)을 유일하게 식별할 수 있는 신분증 번호와 같습니다.
*   **특징:** 중복될 수 없으며, `NULL` 값을 가질 수 없습니다. 테이블당 하나만 설정 가능합니다.

### **② Foreign Key (FK, 외래키)**
*   **역할:** 한 테이블의 컬럼이 다른 테이블의 PK를 참조하도록 연결합니다.
*   **특징:** 테이블 간의 **관계를 형성**하며, 부모 테이블에 없는 데이터를 자식 테이블에 넣을 수 없게 막아 '데이터 무결성'을 유지합니다.

### **③ Unique Key (고유키)**
*   **역할:** 중복된 값을 허용하지 않습니다.
*   **차이점:** PK와 달리 `NULL` 값을 가질 수 있으며, 한 테이블에 여러 개 설정할 수 있습니다. (예: 이메일, 전화번호)

---

## 2. 테이블 생성, 수정, 삭제 (DDL)

```sql
-- 1. 테이블 생성 (CREATE)
CREATE TABLE users (
    id BIGINT AUTO_INCREMENT PRIMARY KEY, -- PK 설정
    email VARCHAR(255) NOT NULL UNIQUE,    -- Unique 설정
    name VARCHAR(100) NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE orders (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id BIGINT,                        -- FK로 사용할 컬럼
    order_date DATE,
    -- FK 제약 조건 추가: orders의 user_id는 users의 id를 참조한다
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);

-- 2. 테이블 수정 (ALTER)
ALTER TABLE users ADD COLUMN phone VARCHAR(20); -- 컬럼 추가
ALTER TABLE users MODIFY COLUMN name VARCHAR(200); -- 컬럼 수정

-- 3. 테이블 삭제 (DROP)
DROP TABLE orders; -- 테이블 통째로 삭제
```

## 3. 테이블 간의 관계 (Relationships) 상세 분석

현실 세계의 복잡한 데이터 연결 구조를 관계형 데이터베이스에 녹여내는 3가지 방식입니다.

### **① 1:N 관계 (일대다, One-to-Many) - 가장 보편적**
한 쪽 엔티티가 다른 쪽 엔티티를 여러 개 가질 수 있는 구조입니다.
*   **예시:** 부서(1) : 직원(N), 게시글(1) : 댓글(N), 고객(1) : 주문(N)
*   **구현 방법:** **'N' 쪽 테이블에 '1' 쪽의 PK를 FK(외래키)로 추가**합니다.
    *   *생각해보기:* 직원 테이블에 '부서 ID'가 적혀 있어야, 이 직원이 어느 부서 소속인지 알 수 있습니다.

### **② 1:1 관계 (일대일, One-to-One)**
두 테이블의 행이 서로 정확히 하나씩만 대응되는 구조입니다.
*   **예시:** 사용자(1) : 사용자 상세 프로필(1), 국가(1) : 수도(1)
*   **구현 방법:** 어느 한 쪽 테이블에 상대방의 PK를 FK로 넣습니다. (이때 해당 FK는 **UNIQUE** 제약조건이 필수입니다.)
*   **사용 이유:** 보안상 민감한 정보를 분리하거나, 테이블이 너무 커져서 성능 관리를 위해 쪼개야 할 때 사용합니다.

### **③ N:M 관계 (다대다, Many-to-Many)**
양쪽 엔티티가 서로를 여러 개 가질 수 있는 구조입니다.
*   **예시:** 학생(N) : 강의(M), 포스트(N) : 태그(M)
*   **중요:** RDBMS에서는 두 테이블만으로는 N:M을 직접 구현할 수 없습니다. 반드시 **'연결 테이블(Mapping/Junction Table)'**이 중간에 필요합니다.
*   **구현 방법:** `학생`과 `강의` 사이에 `수강신청`이라는 테이블을 만들고, 여기에 `학생_ID`와 `강의_ID`를 각각 FK로 담습니다.
    *   *결과:* 연결 테이블을 기준으로 보면 `학생 : 수강신청`은 1:N, `강의 : 수강신청`도 1:N 관계가 되어 다대다를 해결합니다.

## 4. 관계 설정의 핵심: 외래키 참조 무결성 (Cascading)

부모 데이터(1)가 수정되거나 삭제될 때, 연결된 자식 데이터(N)를 어떻게 처리할지 결정하는 옵션입니다.

*   **ON DELETE CASCADE:** 부모를 지우면 자식도 **함께 삭제**됩니다. (예: 게시글 삭제 시 댓글도 삭제)
*   **ON DELETE SET NULL:** 부모를 지우면 자식의 FK 값을 **NULL로 변경**합니다. (예: 부서 삭제 시 직원은 남겨두고 부서 정보만 비움)
*   **ON DELETE RESTRICT:** 자식이 남아있으면 부모를 **지울 수 없게 막습니다.** (기본값, 데이터 보호용)

## 5. 용어 정리 Q&A

*   **Q: PK는 꼭 숫자로 해야 하나요?**
    *   A: 주로 `AUTO_INCREMENT` 숫자를 쓰지만, UUID 같은 고유 문자열을 쓰기도 합니다. 다만 인덱스 성능상 숫자가 유리합니다.
*   **Q: FK가 없어도 JOIN은 되는데 왜 FK를 쓰나요?**
    *   A: FK는 '데이터의 안전망'입니다. 실수로 부모 없는 데이터를 만들거나, 주문 내역이 있는데 사용자를 지워버리는 불상사를 막아줍니다.
*   **Q: INDEX와 UNIQUE의 차이는?**
    *   A: UNIQUE는 중복 방지가 목적이고, INDEX는 조회 속도 향상이 목적입니다. (보통 UNIQUE 설정 시 자동으로 INDEX가 생성됩니다.)
