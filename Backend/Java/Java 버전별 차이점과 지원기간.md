# Java 버전별 주요 특징 및 지원 기간 (LTS 중심)

Java는 현재 2년 주기로 LTS(Long-Term Support) 버전을 출시하며, 현대적인 프로그래밍 언어로 지속적으로 진화하고 있습니다. 이 글에서는 실무에서 주로 사용되는 LTS 버전들의 차이점과 지원 로드맵을 정리합니다.

---

## 1. Java LTS 버전 지원 로드맵

Java의 각 버전은 벤더(Oracle, Red Hat, Eclipse 등)에 따라 지원 기간이 다르지만, 일반적으로 Oracle의 가이드라인을 기준으로 합니다.

| 버전 | 출시일 | 프리미어 지원 종료 | 연장 지원 종료 | 비고 |
| :--- | :--- | :--- | :--- | :--- |
| **Java 8** | 2014년 3월 | 2022년 3월 | **2030년 12월** | 여전히 현업 점유율 높음 |
| **Java 11** | 2018년 9월 | 2023년 9월 | **2032년 1월** | 모듈 시스템 안정화 |
| **Java 17** | 2021년 9월 | 2026년 9월 | 2029년 9월 | 현재 가장 권장되는 버전 |
| **Java 21** | 2023년 9월 | 2028년 9월 | 2031년 9월 | 가상 스레드(Loom) 도입 |
| **Java 25** | 2025년 9월 | 2030년 9월 | 2033년 9월 | 최신 LTS 버전 |

> **Tip:** 신규 프로젝트라면 최소 **Java 17** 이상을 선택하는 것이 보안 및 성능 면에서 유리하며, 최신 기능을 위해 **Java 21**이나 **Java 25**를 고려해 보세요.

---

## 2. LTS 버전별 주요 변화 (What's New?)

### **Java 8: 함수형 프로그래밍의 시작**
*   **Lambda Expressions & Streams:** 자바에 함수형 패러다임을 도입하여 코드를 간결하게 작성할 수 있게 되었습니다.
*   **Optional Class:** `NullPointerException` 방지를 위한 표준화된 객체 핸들링.
*   **New Date/Time API:** 기존의 문제 많은 `Date`, `Calendar`를 대체하는 `java.time` 패키지 도입.

### **Java 11: 현대화와 정돈**
*   **`var` 키워드:** 지역 변수에 대한 타입 추론을 지원하여 가독성을 높였습니다.
*   **HTTP Client API:** 표준화된 비동기 HTTP 통신을 지원하는 현대적인 클라이언트 API.
*   **Oracle JDK 유료화 대응:** OpenJDK 중심의 생태계로 전환되는 시점.

### **Java 17: 생산성 향상의 완성**
*   **Sealed Classes:** 상속 및 구현이 가능한 클래스를 제한하여 코드의 안전성을 강화.
*   **Records:** 데이터 중심의 클래스(DTO 등)를 작성할 때 보일러플레이트 코드를 획기적으로 줄여줍니다.
*   **Pattern Matching for `instanceof`:** 타입 체크와 캐스팅을 한 번에 처리.

### **Java 21: 동시성 혁명**
*   **Virtual Threads (Project Loom):** 기존의 무거운 OS 스레드 대신 가벼운 가상 스레드를 사용하여 수백만 개의 동시 요청을 효율적으로 처리.
*   **Sequenced Collections:** 리스트와 셋에서 첫 번째/마지막 요소에 접근하는 표준화된 방식 제공.
*   **Record Patterns:** 레코드 객체의 데이터를 쉽게 분해하고 추출.

### **Java 25: 성능과 AI 시대의 완성**
*   **Compact Object Headers:** 객체 헤더 크기를 줄여 메모리 사용량 최적화 및 성능 향상.
*   **Flexible Constructor Bodies:** `super()` 호출 전 로직 실행이 가능해져 생성자 구현이 유연해짐.
*   **Structured Concurrency:** 여러 비동기 작업을 하나의 작업 단위처럼 관리하여 안정성 강화.
*   **Vector API:** 고성능 수치 연산 및 AI 관련 계산 최적화 지원.

## 3. Java 8에서 상위 버전으로의 전환이 어려운 이유

단순히 버전 숫자만 올리는 것이 아니라, Java 9에서 도입된 **구조적 변화**가 가장 큰 장벽입니다.

### **① Project Jigsaw (모듈 시스템) 도입**
Java 9부터 도입된 모듈 시스템은 JDK 내부의 캡슐화를 강화했습니다. 많은 오픈소스 라이브러리가 기존의 내부 API에 의존하고 있어, 라이브러리 업데이트가 동반되어야 합니다.

### **② Java EE 모듈의 제거 (Java 11)**
JDK에 포함되어 있던 JAXB, JAX-WS 등 Java EE 관련 모듈들이 삭제되어, 이제는 외부 라이브러리로 별도 관리해야 합니다.

### **③ 가비지 컬렉터(GC)의 변화**
기본 GC가 Parallel에서 G1 GC로 변경되면서 성능 특성이 달라졌으므로, 세밀한 튜닝이 다시 필요할 수 있습니다.

---

## 4. 어떤 버전을 선택해야 할까?

*   **안정성이 최우선인 기존 서비스:** Java 11 또는 17
*   **신규 서비스 개발:** **Java 17** (Spring Boot 3.x 최소 사양) 혹은 **Java 21**
*   **최첨단 성능과 AI 기능 활용:** **Java 25**

Java는 더 이상 "느리고 무거운" 언어가 아닙니다. 25 버전 이후의 Java는 현대적인 기능들을 대거 포함하며 다시 한번 대세 언어로서의 자리를 굳히고 있습니다.
